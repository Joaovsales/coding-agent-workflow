# Spec: Trim Session-Start Token Footprint

## Behavior

Reduce the project-controlled portion of context loaded at every Claude Code session start by removing duplicated and inactive content from `CLAUDE.md` and `.claude/hooks/session-start.sh`, **without changing any workflow capability** (deployment verification still discoverable, skill list still surfaced, hooks still fire correctly, all skills still invocable).

The optimization is purely textual / structural — no skill, agent, hook, or routing logic is removed.

## Inputs

Existing files at HEAD on branch `claude/analyze-token-usage-LS6wm`:

- `CLAUDE.md` (14,254 bytes / ~3,750 tokens)
- `.claude/hooks/session-start.sh` (8,746 bytes; produces 2,802 bytes of output / ~740 tokens per session)
- `.claude/deployments/README.md` (existing adapter contract reference)

## Outputs

### Change 1 — Move inactive deployment schema example out of `CLAUDE.md`

Source: the `## Deployment Verification — Schema Reference (Inactive Example)` section (2,789 bytes / ~735 tokens) at the bottom of `CLAUDE.md`.

Action:
- **Move** the section body — preamble warning, indented code block, schema rules, and config block explanation — into `.claude/deployments/README.md` under a new heading `## CLAUDE.md Deployment Targets — Schema Reference for Downstream Projects` (or similar, appended at the end of that file).
- **Replace** the section in `CLAUDE.md` with a 2-line pointer:

  ```markdown
  ## Deployment Verification

  This template repo has no active deployment targets. Downstream projects: run `/setup-deployment` to populate a `## Deployment Targets` section here. Schema reference: `.claude/deployments/README.md`.
  ```

  The replacement section's heading **must not** match the regex `^## Deployment Targets[[:space:]]*$` so that `/verify-deployment` and `session-start.sh` continue to treat the template repo as inactive.

Net token impact: ~735 tokens saved per session.

### Change 2 — Curate the `SKILLS AVAILABLE` block in `session-start.sh` to a starter trio

Source: lines 151–169 of `.claude/hooks/session-start.sh` print 15 skill names + descriptions (932 bytes / ~245 tokens of session-start hook output).

Action:
- Replace the hardcoded list of 15 skill lines with **3 starter-trio lines**:
  - `/plan        — Spec + task breakdown`
  - `/build       — Autonomous TDD execution`
  - `/wrap-up-session — Close session: review, test, push`
- Add a single trailing line: `(type / to see all skills)`
- Keep the `SKILLS AVAILABLE` header and divider — they orient the agent to the section.

Net token impact: ~180 tokens saved per session (~932 - ~250 retained for trio header + 4 lines).

### Change 3 — None for skill descriptions

Per user direction (Q3), skill frontmatter `description:` fields are **not** modified. Zero changes to `.claude/skills/*/SKILL.md` files.

## Edge Cases

| Case | Expected behavior |
|---|---|
| `session-start.sh` runs in a downstream project that has populated `## Deployment Targets` | Behavior unchanged — Change 1 only touches the template repo's `CLAUDE.md`; downstream `CLAUDE.md` is generated by `/setup-deployment` and is independent. |
| User runs `/setup-deployment` after Change 1 lands | Skill must still be able to find the schema reference. The replacement pointer in `CLAUDE.md` names the new location explicitly. |
| `/verify-deployment` parses `CLAUDE.md` looking for the strict regex `^## Deployment Targets[[:space:]]*$` | Replacement heading is `## Deployment Verification` (not the strict literal) → continues to match zero. Verified by `grep -E '^## Deployment Targets[[:space:]]*$' CLAUDE.md` returning empty. |
| `session-start.sh`'s deploy-signal nudge logic (lines 75–93) | Unchanged. Logic only inspects `CLAUDE.md` for the strict regex; replacement heading does not match → nudge still fires correctly when signal files exist. |
| User unfamiliar with the workflow opens a fresh session | Sees the trio (`/plan`, `/build`, `/wrap-up-session`) + the "type / to see all skills" pointer. The harness's skill auto-listing surfaces the rest. |
| `.claude/deployments/README.md` already contains a heading collision with the new appended section | Append uses a unique heading; verify with `grep` before writing. If collision, fail loudly rather than silently overwriting. |
