# /start-qa — Start QA Session

Restart the application, launch a browser for manual QA as fast as possible, and run health/smoke checks in the background.

---

## Step 1 — Load or Build Project Config

Check if `.claude/qa-config.md` exists in the project root.

### If the file EXISTS — load it and skip to Step 2

Read `.claude/qa-config.md` and use its values for all subsequent steps. Do not re-scan the project.

### If the file DOES NOT EXIST — run one-time discovery

Scan the project's configuration files to determine:

1. **How to start the app** — check in order:
   - `Makefile` / `makefile` (e.g., `make up`, `make dev`, `make start`)
   - `docker-compose.yml` / `compose.yml` (e.g., `docker compose --env-file .env up -d`)
   - `package.json` scripts (e.g., `npm run dev`, `npm start`)
   - `Procfile`, `pyproject.toml`, or framework-specific configs
   - `README.md` for documented startup instructions

2. **How to stop the app** — corresponding teardown command (e.g., `make down`, `docker compose down`, process kill)

3. **What URL the app runs on** — check:
   - Docker compose port mappings
   - Vite/webpack/Next config files
   - `.env` / `.env.local` for `PORT` or `HOST`
   - Framework defaults (Vite: 5173, Next.js: 3000, FastAPI: 8000, Rails: 3000)

4. **How to liveness-check the app** — e.g., `curl -sf <url>` or container/process status check

5. **Health endpoint** — look for `/health`, `/api/health`, `/healthz`, or equivalent

6. **Pre-QA test command** — check for:
   - Makefile targets: `pre-QA-tests`, `test-smoke`, `test-critical`, `test-mvp`, `test-health`
   - Scripts: `scripts/pre_qa*.sh`, `scripts/smoke*.sh`
   - If none exist, note "none found" — the loop will fall back to the liveness check

7. **How to access runtime logs** — e.g., `docker compose logs -f`, log file path, or terminal process

Once discovered, **write `.claude/qa-config.md`** with this exact structure:

```markdown
# QA Config — [Project Name]
> Auto-generated by /start-qa. Edit this file to override any value.

## Startup
- **Start command:** `<command>`
- **Stop command:** `<command>`
- **App URL:** `<url>`
- **Liveness check:** `<command or curl>`

## Health
- **Health endpoint:** `<url or "none">`
- **Pre-QA test command:** `<command or "none">`

## Logs
- **Log command:** `<command>`

## Notes
<Any project-specific quirks discovered during setup, e.g. "requires .env to exist", "run migrations before starting", etc.>
```

Confirm to the user: "First run — project config saved to `.claude/qa-config.md`. Edit that file any time to update startup behavior."

---

## Step 2 — Restart the Application

Using the config from Step 1:

1. Run the stop command (if applicable)
2. Run the start command
3. Wait a few seconds, then run the liveness check

If the liveness check fails, inspect logs using the log command, diagnose the root cause, fix it, and retry. Loop until the app is reachable.

---

## Step 3 — Launch Browser Immediately

As soon as the liveness check passes, launch the browser **without waiting for pre-QA tests**:

```
/chrome
```

Navigate to the app URL from the config. The `/chrome` session captures **console logs** and **network requests** automatically.

**Report QA ready immediately:**

```
QA Session Ready
─────────────────────────────────────
App Status:    Running
Browser:       Launched at [URL]
Log Monitoring:
  - Console logs: Active
  - Network logs: Active

Background: Pre-QA smoke tests are running via /loop.
            You'll be notified only if something fails.

You can start manual QA now.
─────────────────────────────────────
```

---

## Step 4 — Background Health & Smoke Tests via /loop

Immediately after launching the browser, start a background loop using the pre-QA test command from config:

```
/loop 2m Read .claude/qa-config.md to get the pre-QA test command and log command. Run the pre-QA test command. If it fails or any infrastructure issue is detected: read logs using the log command, diagnose the root cause, fix it, and re-run. Report to the user only when something fails or when all checks pass clean. Do not interrupt QA for passing tests.
```

---

## During QA (Ongoing)

While the user performs manual QA:
- Keep the browser session and log monitoring active
- When the user reports an issue, check captured console errors and failed network requests
- Access backend logs using the log command from config
- Suggest fixes or file bugs in `tasks/bugs.md` as issues are discovered
